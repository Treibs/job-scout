job/scout · how it works

One engine · two homes

The same pipeline runs either on your machine (state on disk, can score key-free via Claude Code) or on GitHub Actions (state in a private cache, scores with an API-key secret). Each step below is its own small diagram — read them top to bottom.

Part 1

On your machine

local-first · runs offline-ish, key optional

1.1 The daily scan — how a listing becomes a scored match

A linear pipeline. Each new role is gathered, cleaned, de-duplicated, enriched, then scored against your résumé.

flowchart LR
  saB["Job boards and ATS
Indeed - LinkedIn - Workday"] --> saN["Normalize
one common shape"]
  saN --> saD["Dedupe
skip ones seen before"]
  saD --> saE["Enrich
full description for top matches"]
  saE --> saS["Score
fit vs your resume"]
  saS --> saC[("output/jobs.csv
your tracker")]
  saL{{"LLM provider
MiniMax key OR Claude CLI"}} -. "does the scoring" .-> saS

1.2 The news board — how headlines become a relevant feed

Same idea, separate pipeline. Free sources in; a scored, summarized feed out. No API key needed to fetch.

flowchart LR
  nwQ["Google News - GDELT
free, no key"] --> nwD["Dedupe"]
  nwD --> nwS["Score relevance
+ short summary"]
  nwS --> nwJ[("state/news.json
the feed")]
  nwL{{"LLM provider"}} -. "summarizes" .-> nwS

1.3 Reviewing — the two dashboards

A tiny local server renders both feeds at localhost:8765. Your triage saves straight back into the files.

  rvC[("jobs.csv")] --> rvSV["Dashboards
serve.py - localhost:8765"]
  rvNJ[("news.json")] --> rvSV
  rvSV --> rvY(["You
filter - status - notes - vote"])
  rvY -- "saved back" --> rvC
  rvCN[("LinkedIn connections

1.4 The feedback loop — why it gets smarter

The cycle that closes it. Your signals feed a strategist that re-tunes what gets searched next — then it loops.

flowchart LR
  fbA["1 - Scan
new roles only"] --> fbB["2 - Score + summarize"]
  fbB --> fbC["3 - You triage
interested - applied - vote"]
  fbC --> fbD[("4 - Ledger + feedback
discovery.json + jobs.csv")]
  fbD --> fbE["5 - Strategist
every 3 days - resume guardrail"]
  fbE --> fbF[("6 - discovery_additions.yaml
new keywords / companies")]
  fbF --> fbA

1.5 What lives on disk — the “memory”

Persistence is what makes every run incremental. Re-running scripts/setup.py just picks up from here.

output/jobs.csv
Your tracker — every role plus your statuses & notes.
state/news.json
The news feed cache (keeps your 👍/👎).
state/seen_hashes.json
What's already been seen — powers cross-run dedupe.
state/discovery.json
The ledger — what each keyword/company yields.
config/discovery_additions.yaml
The strategist's proposed keywords & companies.
resume/resume.md
What scoring compares each role against. private
data/linkedin_connections.csv
Your network, for the 🤝 overlay. private · optional
Part 2

On GitHub Actions

hosted · public repo · no machine to keep on

2.1 What starts a run

Three triggers, all landing on a fresh throwaway VM. No personal data ever lives in the public repo.

flowchart LR
  tgT1["Daily cron
daily.yml"] --> tgGH["GitHub Actions"]
  tgT2["Every 3 days
strategist.yml"] --> tgGH
  tgT3["Manual / API
on_demand.yml"] --> tgGH
  tgGH --> tgR["Fresh runner
a clean VM each run"]

2.2 What one run does

A clean VM has no memory, so it restores state at the start and saves it at the end — and ships results out.

flowchart LR
  rnCO["Checkout code"] --> rnPIP["pip install"]
  rnPIP --> rnRST["Restore state
from cache"]
  rnRST --> rnRUN["Run
scan + news"]
  rnRUN --> rnSAV["Save state
to cache"]
  rnRUN --> rnART["Upload results
as artifact"]

2.3 Where memory and the key come from

Memory is the cache (restored in, saved out — never committed, since it holds your targets). The key is a secret. Results leave as a download or a Sheet.

flowchart LR
  ckCACHE[("actions/cache
your state - private")] -- "restore" --> ckRUN["The run"]
  ckRUN -- "save" --> ckCACHE
  ckSEC["Secrets
ANTHROPIC_API_KEY"] -- "the key" --> ckRUN
  ckRUN --> ckOUT["Download artifact
or write to Google Sheet"]

2.4 How you actually see the results — and what's clickable

The key difference. Local runs a live server you click and it saves. GitHub has no server — you get a static file to download or a Sheet, so filtering works but the file can't save your triage back.

flowchart LR
  vwRUN["A finished run produces
jobs.html - news.html - jobs.csv"] --> vwL["LOCAL
serve.py - live at localhost:8765"]
  vwRUN --> vwA["GITHUB
download artifact, open jobs.html"]
  vwRUN --> vwS["GITHUB
Google Sheet (optional)"]
  vwL --> vwLY(["You: filter + triage
clicks SAVE back to the CSV"])
  vwA --> vwAY(["You: filter only
static file - no save-back"])
  vwS --> vwSY(["You: triage in the Sheet"])
